# フィードバック：OAuth承認の失効による403（2026-06-04発生・06-10復旧）

## 現象

- 2026-06-04 未明から、アプリの保存が「ネットワークエラー」で失敗
- GitHub Actions の keep-alive ping も 6/4 02:17 JST から53回連続失敗
- Google からの失敗通知メール（毎日3時すぎ）に決定的な証拠：
  > Function: `keepAlive` / Error: **Authorization is required to perform that action.**
- **keepAlive トリガー自体が認証エラーで失敗** ＝ スクリプト全体の OAuth 承認が失効
- 同時期に別スクリプト「情報収集_AI」も同種の認証エラーで失敗していた（遅くとも5/27から）

## 5/18 の「休眠403」との違い

| | 休眠403（5/18） | 承認失効（6/4） |
|---|---|---|
| keepAliveトリガー | 正常動作 | **トリガー自体が認証エラー** |
| 復旧方法 | testAuth を1回実行（ダイアログなし） | testAuth 実行＋**「許可」ダイアログで再承認** |
| keep-aliveで予防 | できる | **できない**（承認が消えればトリガーも死ぬ） |

## なぜなぜ分析（5回）

1. **なぜ保存に失敗した？** → GAS が 403 を返したため
2. **なぜ 403？** → スクリプトの OAuth 承認が失効していたため
3. **なぜ失効した？** → Google 側でスクリプトへのアクセス権が取り消されたため
4. **なぜ取り消された？（未確定・要調査）** → 候補：(a) パスワード変更・セキュリティ診断での接続一括解除 (b) Google の未検証アプリへの自動失効 (c) 警告メールからの誤操作。→ [接続管理ページ](https://myaccount.google.com/connections) の履歴で確認する
5. **なぜ6日間気づけなかった？** → keep-alive の失敗が GitHub 上にしか出ず、普段見る Chatwork に届いていなかったため

## 根本原因

**「keep-alive で休眠は防げるが、承認失効は防げない」構造**に対し、失効を即座に検知・通知する仕組みがなかった。
OAuth の「許可」はプログラムから自動で押せない（Googleのセキュリティ仕様）ため、対策の本質は「失効ゼロ」ではなく**「検知1時間以内・復旧1分」**にすること。

## 実施した対策（2026-06-10）

| # | 対策 | 実装 |
|---|------|------|
| 1 | keep-alive 失敗→Chatwork 即通知 | keepalive.yml に通知ステップ追加。成功⇄失敗の**変化時のみ**通知（毎時スパム防止）。復旧時は「復旧しました」通知 |
| 2 | アプリ起動時の疎通チェック | index.html に `checkGasHealth()`。起動時 ping 失敗で復旧手順つき警告を常時表示 |
| 3 | 復旧手順の1タップ化 | Chatwork 通知にスプレッドシートURL＋「拡張機能→Apps Script→testAuth→許可」手順を埋め込み |

### 必要なセットアップ（GitHub側・手動）

リポジトリ Settings → Secrets and variables → Actions に以下を登録：
- `CHATWORK_API_TOKEN`（情報収集_AI のスクリプトプロパティと同じ値）
- `CHATWORK_ROOM_ID`（同上）

※未登録の間は通知ステップがスキップされるだけで ping 自体は動く

## 復旧手順（再発時はこれだけ）

1. スプレッドシート「読書記録」を開く
2. 拡張機能 → Apps Script
3. 関数 `testAuth` を選んで実行 → 「許可」をクリック

## 教訓

- **keep-alive は休眠対策であって承認失効対策ではない。** 失効はトリガーごと殺すので、外形監視（外部からのping）＋普段見るチャネルへの通知が必須
- **「ネットワークエラー」の裏に3種類の原因がある**：①本当のネットワーク問題 ②GASの休眠 ③OAuth承認の失効。Googleの失敗通知メール（keepAliveのエラー内容）で③かどうか即判別できる
