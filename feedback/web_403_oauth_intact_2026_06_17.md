# フィードバック：Web 403 だが OAuth は生きている停止（2026-06-17 発生）

## 現象（確定事実）

- 2026-06-17 夕方（JST）、アプリの保存が失敗。keep-alive ping が失敗に転じた。
  - 最後の成功：6/17 05:15 UTC（14:15 JST）
  - 最初の失敗：6/17 10:06 UTC（19:06 JST）以降、連続失敗
- GitHub Actions のエラー注釈で **HTTP 403** を確定（curl はURLに到達済み＝ネットワーク/URL不良ではない）。
- デプロイ設定（スクショで確認）：**アクセス=全員／実行=自分（honma0728@gmail.com）** ＝設定は正しい。
- アクティブなデプロイが「無題」3つ並んでいた（呼び出し対象が曖昧）。

## 前回（6/4 OAuth失効）との決定的な違い

| 項目 | 6/4 OAuth失効 | 6/17 今回 |
|---|---|---|
| 匿名Web ping | 403 | 403（同じ） |
| 内部トリガー keepAlive（オーナー実行） | ❌ 失敗 | ✅ **成功** |
| Google失敗通知メール（読書記録） | 連日着信 | **6/10を最後に来ていない** |

→ Gmail で `noreply-apps-scripts-notifications@google.com` の「読書記録」失敗メールが 6/10 で止まっていることを確認。
6/17 18:00 UTC のオーナー実行トリガーは成功（メールなし）＝**スクリプトのOAuthは生きている**のに、同時間帯の匿名Webだけ403。
**よって今回はOAuth失効ではなく、Webアプリのデプロイ／アクセス権レイヤーの問題。**

## なぜなぜ分析（5回）

1. なぜ保存できない？ → GAS Webエンドポイントが 403
2. なぜ403？ → Googleが匿名アクセスを拒否（URL到達済み＝NW/URL不良ではない）
3. なぜ匿名だけ拒否？ → オーナー実行は成功 ＝ OAuth認可ではなくWebデプロイのアクセス層で弾かれている
4. なぜアクセス層で弾かれる？（**最終原因は未確認**）→ アクセス=全員でも古いデプロイが403を返す事象。Apps Scriptの既知挙動で、標準対処は「新バージョンで再デプロイ＋URL貼り替え」。Workspace制限(c)は @gmail.com 個人アカウントのため否定、アクセス権変更(a)も「全員」表示のため否定。
5. なぜ取り違えやすい？ → 6/10の通知が「403＝OAuth失効、testAuthで復旧」と断定していたため。今回testAuthでは直らない。

## 対処

- **復旧（適用予定・効果は要確認）**：デプロイを管理→編集→バージョン=新バージョン→デプロイ→新exec URLを (a)アプリのGAS URL設定 (b)GitHubのGAS_URLシークレット の両方へ貼り替え→余分な「無題」デプロイはアーカイブで1つに集約。
- **再発防止（実装済み・push済み 9c679b6）**：keepalive.yml の失敗通知に **HTTPステータスを含め、ステータス別に疑う原因と最初に試す対処を出し分け**。401/403はまず再デプロイ→ダメならtestAuth、の順に案内。

## 教訓

- **「403＝OAuth失効」と決め打ちしない。** keepAliveトリガーの成否（＝Google失敗メールの有無）で「OAuth失効」か「Webデプロイ問題」かを切り分けられる。トリガーが生きているなら testAuth では直らない。
- アクティブなデプロイは1つに集約する（複数あると呼び出し対象が曖昧になり事故源）。
